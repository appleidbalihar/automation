# Slack Bot Integration — Operations Runbook

## Production Setup

Slack Phase 1 is bot-first. Users talk to RapidRAG through a Slack bot DM or `/kb`; a Slack channel is not required.

## Public Slack URLs

Development:

```text
OAuth Redirect URL: https://dev.eclassmanager.com/rapidrag/api/slack/oauth/callback
Event Subscriptions Request URL: https://dev.eclassmanager.com/rapidrag/api/slack/events
Slash Command /kb Request URL: https://dev.eclassmanager.com/rapidrag/api/slack/events
```

Production:

```text
OAuth Redirect URL: https://theaitools.ca/rapidrag/api/slack/oauth/callback
Event Subscriptions Request URL: https://theaitools.ca/rapidrag/api/slack/events
Slash Command /kb Request URL: https://theaitools.ca/rapidrag/api/slack/events
```

Manual customer-owned bots use:

```text
https://<domain>/rapidrag/api/slack/events/<deploymentId>
```

### How to create the platform RapidRAG Bot

1. Go to `https://api.slack.com/apps`.
2. Click `Create New App`.
3. Choose `From scratch`.
4. App name: `RapidRAG Bot`.
5. Select the platform/operator workspace used to own the app.
6. Configure OAuth redirect URL:

```text
https://<production-domain>/rapidrag/api/slack/oauth/callback
```

7. Go to `OAuth & Permissions` -> `Scopes` -> `Bot Token Scopes` and add:

```text
chat:write
commands
im:history
```

Do not use `App-Level Tokens` for this flow. App-level tokens are for app/platform features such as Socket Mode; RapidRAG needs the Bot User OAuth Token generated by OAuth install.

8. Go to `App Home`, turn on the `Messages Tab`, and check:

```text
Allow users to send Slash commands and messages from the messages tab
```

This is required so users can send DMs to the bot from the Slack app messages tab.

9. Configure Event Subscriptions request URL:

```text
https://<production-domain>/rapidrag/api/slack/events
```

10. Wait until Slack shows `Verified`.
11. Under `Subscribe to bot events`, click `Add Bot User Event`.
12. Subscribe to bot event:

```text
message.im
```

13. Click `Save Changes`.
14. Configure slash command `/kb` request URL:

```text
https://<production-domain>/rapidrag/api/slack/events
```

Slash command fields:

```text
Command: /kb
Short description: Query RapidRAG knowledge bases
Usage hint: [list | use <name> | all | status | help]
Escape channels, users, and links sent to your app: leave unchecked
```

`/kb` and its subcommands are RapidRAG platform commands. All RapidRAG Bot installs and customer-owned bots use the same command contract. If the platform command set changes later, update this runbook, the Chat Channels wizard, and the Slack app slash-command usage hint together.

15. Install the app to the workspace.
16. Copy `Client ID`, `Client Secret`, and `Signing Secret` from `Basic Information` -> `App Credentials`.
17. Seed Slack app credentials in Vault:

```text
platform/global/slack/oauth/client_id
platform/global/slack/oauth/client_secret
platform/global/slack/oauth/signing_secret
```

## User Setup Paths

Recommended path:

1. User opens Chat Channels.
2. User clicks `Add RapidRAG Bot to Slack`.
3. Slack OAuth fills workspace details automatically.
4. User maps one or more KBs.
5. User activates the deployment.
6. User opens a DM with RapidRAG Bot and asks a question.

Advanced path:

1. User creates their own Slack app, for example `eclass-bot`.
2. User enables `App Home` -> `Messages Tab` and checks `Allow users to send Slash commands and messages from the messages tab`.
3. User configures Event Subscriptions with the deployment-specific URL shown in RapidRAG.
4. User waits for `Verified`, adds the `message.im` bot event under `Subscribe to bot events`, and clicks `Save Changes`.
5. User configures `/kb` with the same deployment-specific URL and the same RapidRAG usage hint.
6. User installs the app.
7. User copies Bot User OAuth Token from `OAuth & Permissions` -> `OAuth Tokens` and Signing Secret from `Basic Information` -> `App Credentials` into RapidRAG.
8. RapidRAG validates and stores the secrets in Vault.

## User Onboarding Message

RapidRAG does not send Slack invites or user notifications in Phase 1. After an admin adds Slack user IDs to an allowlist, the Chat Channels wizard provides a `Copy user instructions` button. The admin sends that text manually by email, Slack, or an internal onboarding process.

Default RapidRAG Bot message:

```text
You have been granted access to the RapidRAG Bot Slack knowledge base bot.

How to use it:
1. Open Slack in the workspace where the bot is installed.
2. Search for "RapidRAG Bot" in Slack.
3. Open a direct message with the bot.
4. Send: /kb list
5. To choose a knowledge base, send: /kb use <name>
6. Then ask your question in the same DM.

If Slack says you are not authorized, reply with your Slack member ID so access can be checked.
To copy your Slack member ID:
Open your Slack profile -> More -> Copy member ID.
```

For customer-owned bots, the message uses the deployment/bot display name instead of `RapidRAG Bot`. No Slack API call is made when this message is copied.

## Public Routes

nginx must pass these paths to api-gateway:

```text
/rapidrag/api/slack/oauth/callback
/rapidrag/api/slack/events
/rapidrag/api/slack/events/<deploymentId>
```

The webhook routes must preserve raw request bodies. Do not log Slack request bodies because slash commands include `response_url`.
Root `/api/...` must not be used on shared domains because it can belong to other applications.

## Rate Limits

Prefer app-level throttling for normal backpressure because it can still acknowledge Slack and send a friendly message:

```text
I'm receiving too many requests right now. Please try again in a minute.
```

Use nginx `limit_req` only for abusive floods after staging validation. nginx rejection happens before the app can send a Slack reply, and Slack may retry non-2xx responses.

Track deployed values for:

```text
SLACK_WEBHOOK_RATE_LIMIT_PER_MINUTE
SLACK_WEBHOOK_RATE_LIMIT_BURST
SLACK_DIFY_CONCURRENCY_PER_DEPLOYMENT
```

## Smoke Test

1. Open Chat Channels.
2. Click `Add RapidRAG Bot to Slack` and complete OAuth.
3. Select at least one KB.
4. Activate deployment without selecting a channel.
5. In Slack, open a DM with RapidRAG Bot.
6. Run:

```text
/kb list
/kb use 1
```

7. Ask a KB question in the DM.
8. Confirm the Chat Channels history drawer shows the conversation.

## Troubleshooting

- Invalid Slack signature: check signing secret, raw body forwarding, and timestamp skew.
- OAuth callback rejected: check client ID/secret, redirect URL, and Redis availability for state nonce.
- OAuth returns `bad_redirect_uri`: confirm Slack has the exact `/rapidrag/api/slack/oauth/callback` URL, api-gateway does not follow workflow-service redirects internally, and workflow-service reuses the original `redirect_uri` from signed OAuth state during `oauth.v2.access`.
- Browser shows Cloudflare `502 Bad Gateway` immediately after Slack authorization, but the deployment still connects: check api-gateway redirect forwarding. The proxy must use `redirect: "manual"` and forward workflow-service redirects with `reply.redirect(location, response.status)`. The reversed argument order creates an invalid `Location: 302` response that Cloudflare displays as 502 even though OAuth already succeeded.
- Browser shows `Route GET:/rapidrag/chat-channels... not found` after Slack OAuth: api-gateway is following the workflow-service redirect instead of returning the `Location` header to the browser.
- Sending messages to bot is turned off: enable `App Home` -> `Messages Tab` and check `Allow users to send Slash commands and messages from the messages tab`.
- Bot does not reply in DM: confirm deployment is active, workspace ID matches, `message.im` is subscribed, bot token is stored, Messages Tab is enabled, and mapped KBs have Dify app credentials.
- Duplicate replies: check Redis connectivity and dedup keys.
- Dify outage: users receive a safe Slack error message; repeated failures mark the deployment `error`.

## Rollback

1. Deactivate affected Slack deployments in Chat Channels.
2. Disable Slack app Event Subscriptions or slash command if needed.
3. Temporarily remove nginx Slack public routes only for severe abuse or security incidents.
4. Keep database rollback aligned with the Prisma migration applied for Chat Channels.
