## Pre-flight Checklist

- [ ] Production server has Docker + Docker Compose v2 installed
- [ ] `theaitools.ca` DNS A record points to server's public IP
- [ ] Ports 80 and 443 are open in the firewall; port 3443 is NOT public
- [ ] Certbot installed: `sudo yum install certbot python3-certbot-nginx` (RHEL) or `sudo apt install certbot python3-certbot-nginx` (Ubuntu)
- [ ] Server has ≥ 8 GB RAM and ≥ 100 GB free on `/home`
- [ ] `jq` installed: `sudo yum install jq` or `sudo apt install jq`
- [ ] You have an OpenAI-compatible LLM API key (for the AI Agent Prompt page — Step 10)

## Step 1 — Configure Docker Data Root

> `/opt` and `/` typically have limited space (~48 GB). Store all Docker data on `/home` which has 2.7 TB.

```bash
sudo mkdir -p /home/docker-data

sudo tee /etc/docker/daemon.json > /dev/null <<'EOF'
{
  "data-root": "/home/docker-data",
  "live-restore": false,
  "default-shm-size": "64M",
  "log-driver": "json-file",
  "log-opts": { "max-size": "10m", "max-file": "3" },
  "storage-driver": "overlay2"
}
EOF

sudo systemctl restart docker
docker info | grep "Docker Root Dir"
# Expected: Docker Root Dir: /home/docker-data
```

## Step 4 — Bootstrap Vault and Seed All Secrets

### 4.1 — Mount a tmpfs for the runtime env (one-time OS setup)

```bash
sudo mkdir -p /run/platform-secrets
sudo mount -t tmpfs -o size=4m,noexec,nosuid,nodev tmpfs /run/platform-secrets

# Persist across reboots
echo "tmpfs /run/platform-secrets tmpfs size=4m,noexec,nosuid,nodev 0 0" | sudo tee -a /etc/fstab
```

### 4.2 — Start Vault and wait for bootstrap

```bash
docker compose -f docker-compose.yml -f docker-compose.prod.yml \
  --env-file .env.production up -d vault vault-bootstrap

# Wait ~30 seconds for Vault to initialise and PKI to be bootstrapped
until docker logs $(docker ps -qf name=vault-bootstrap) 2>&1 | grep -q "bootstrap complete\|watch mode enabled"; do
  echo "Waiting for Vault bootstrap..."; sleep 5
done
echo "Vault ready."
```

### 4.3 — Get the root token

```bash
VAULT_DATA_DIR=$(docker volume inspect "$(basename $(pwd))_vault_data" --format '{{.Mountpoint}}')
VAULT_ROOT_TOKEN=$(sudo jq -r '.root_token' "${VAULT_DATA_DIR}/vault-init.json")
echo "Root token: ${VAULT_ROOT_TOKEN}"
```

### 4.4 — Seed all production secrets

```bash
VAULT_ADDR=http://localhost:8200 \
VAULT_TOKEN="${VAULT_ROOT_TOKEN}" \
bash infra/vault/seed-secrets.prod.sh
```

### 4.5 — Verify all secrets are present

```bash
ENVIRONMENT=prod bash scripts/list-secrets.sh
# All 8 paths must show [OK]
```

### 4.6 — Revoke the root token (do this immediately)

```bash
docker exec -e VAULT_TOKEN="${VAULT_ROOT_TOKEN}" \
  $(docker ps -qf name=09_rapidrag-vault-1) \
  vault token revoke "${VAULT_ROOT_TOKEN}"
echo "Root token revoked."
```

### 4.7 — Generate the runtime env file

```bash
ENVIRONMENT=prod \
OUTPUT_FILE=/run/platform-secrets/.env.runtime \
bash scripts/generate-runtime-env.sh
```

## Step 6 — Start the Full Stack

```bash
docker compose -f docker-compose.yml -f docker-compose.prod.yml \
  --env-file .env.production \
  --env-file /run/platform-secrets/.env.runtime \
  up -d

# Delete the runtime env file immediately after (it's in container memory now)
shred -u /run/platform-secrets/.env.runtime

# Verify all containers are up
docker ps --format "table {{.Names}}\t{{.Status}}"
```

**Expected state after a clean start:**
- `db-migrate` — Exited (0) ✓
- `dify-migrate` — Exited (0) ✓
- All other containers — Up ✓

> If `dify-migrate` or `db-migrate` show Exited (1), check their logs:
> ```bash
> docker logs $(docker ps -aqf name=db-migrate)
> docker logs $(docker ps -aqf name=dify-migrate)
> ```
> Common cause: the runtime env file was missing. Re-generate it (Step 4.7) and re-run `up -d`.

## Step 7 — Post-Start: Keycloak Admin User

Create the `platform-admin` user via the Admin REST API (no browser required).

```bash
# 1. Get the Keycloak admin password from Vault
ENVIRONMENT=prod SHOW_VALUES=true PATH_FILTER=infra/keycloak bash scripts/list-secrets.sh
# Note the admin_password value

KC_ADMIN_PASS="<admin_password from above>"
KC_URL="https://localhost:8443"

# 2. Get admin token
TOKEN=$(curl -sk -X POST "${KC_URL}/realms/master/protocol/openid-connect/token" \
  -d "client_id=admin-cli&grant_type=password&username=admin&password=${KC_ADMIN_PASS}" \
  | grep -o '"access_token":"[^"]*"' | cut -d'"' -f4)

# 3. Create platform-admin user
curl -sk -o /dev/null -w "Create user: %{http_code}\n" \
  -X POST "${KC_URL}/admin/realms/automation-platform/users" \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"username":"platform-admin","email":"platform-admin@platform.local","firstName":"Platform","lastName":"Admin","enabled":true,"emailVerified":true}'

# 4. Get user ID
USER_ID=$(curl -sk "${KC_URL}/admin/realms/automation-platform/users?username=platform-admin" \
  -H "Authorization: Bearer ${TOKEN}" \
  | grep -o '"id":"[^"]*"' | head -1 | cut -d'"' -f4)
echo "User ID: ${USER_ID}"

# 5. Set a strong password (change PLATFORM_ADMIN_PASS to something secure)
PLATFORM_ADMIN_PASS=$(openssl rand -base64 20 | tr -d '=+/' | head -c 24)
echo "platform-admin password: ${PLATFORM_ADMIN_PASS}"  # Save this!

curl -sk -o /dev/null -w "Set password: %{http_code}\n" \
  -X PUT "${KC_URL}/admin/realms/automation-platform/users/${USER_ID}/reset-password" \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d "{\"type\":\"password\",\"value\":\"${PLATFORM_ADMIN_PASS}\",\"temporary\":false}"

# 6. Assign admin role
ROLE=$(curl -sk "${KC_URL}/admin/realms/automation-platform/roles/admin" \
  -H "Authorization: Bearer ${TOKEN}")

curl -sk -o /dev/null -w "Assign role: %{http_code}\n" \
  -X POST "${KC_URL}/admin/realms/automation-platform/users/${USER_ID}/role-mappings/realm" \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d "[${ROLE}]"
```

All three commands should print HTTP 201, 204, 204.

## Step 8 — Post-Start: n8n Owner Setup and Workflow Import

> n8n 2.x requires: **create owner → import workflows via CLI → publish via CLI → activate via REST API → restart**. The browser wizard and UI import work but are unreliable for webhooks — always use the CLI approach below.

### 8.1 — Create owner account (one-time)

```bash
curl -s -X POST "http://localhost:5679/rest/owner/setup" \
  -H "Content-Type: application/json" \
  -d '{
    "email": "admin@platform.local",
    "firstName": "Platform",
    "lastName": "Admin",
    "password": "CHOOSE_A_STRONG_PASSWORD",
    "agree": true
  }'
# Expected: HTTP 200 with owner user JSON
```

Save the owner `id` from the response — needed for workflow import.

### 8.2 — Import workflows via CLI

```bash
# Sign in and get session cookie
curl -s -c /tmp/n8n_cookies.txt -X POST "http://localhost:5679/rest/login" \
  -H "Content-Type: application/json" \
  -d '{"emailOrLdapLoginId":"admin@platform.local","password":"YOUR_PASSWORD"}'

# Get owner user ID
OWNER_ID=$(curl -s -b /tmp/n8n_cookies.txt http://localhost:5679/rest/users \
  | grep -o '"id":"[^"]*"' | head -1 | cut -d'"' -f4)
echo "Owner ID: $OWNER_ID"

# Copy workflow templates into the container
docker cp infra/n8n/templates/github-to-dify-sync.json \
  $(docker ps -qf name=n8n-1):/tmp/github-workflow.json
docker cp infra/n8n/templates/gitlab-to-dify-sync.json \
  $(docker ps -qf name=n8n-1):/tmp/gitlab-workflow.json

# Import via CLI (ignore the "Deactivating" warning — that's expected)
docker exec $(docker ps -qf name=n8n-1) \
  n8n import:workflow --input=/tmp/github-workflow.json --userId="$OWNER_ID"
docker exec $(docker ps -qf name=n8n-1) \
  n8n import:workflow --input=/tmp/gitlab-workflow.json --userId="$OWNER_ID"
```

### 8.3 — Publish workflows via CLI

```bash
# Get the IDs of the imported workflows
GH_ID=$(curl -s -b /tmp/n8n_cookies.txt http://localhost:5679/rest/workflows \
  | python3 -c "import sys,json; d=json.load(sys.stdin); \
    print(next(w['id'] for w in d['data'] if 'GitHub' in w['name']))")
GL_ID=$(curl -s -b /tmp/n8n_cookies.txt http://localhost:5679/rest/workflows \
  | python3 -c "import sys,json; d=json.load(sys.stdin); \
    print(next(w['id'] for w in d['data'] if 'GitLab' in w['name']))")

docker exec $(docker ps -qf name=n8n-1) n8n publish:workflow --id="$GH_ID"
docker exec $(docker ps -qf name=n8n-1) n8n publish:workflow --id="$GL_ID"
```

### 8.4 — Activate workflows via REST API

```bash
for id in $GH_ID $GL_ID; do
  curl -s -b /tmp/n8n_cookies.txt -X PATCH "http://localhost:5679/rest/workflows/$id" \
    -H "Content-Type: application/json" -d '{"active":true}' \
    | python3 -c "import sys,json; d=json.load(sys.stdin); \
      print(d['data']['name'], '→ active:', d['data']['active'])"
done
```

### 8.5 — Restart n8n to register webhooks

```bash
docker compose -f docker-compose.yml -f docker-compose.prod.yml restart n8n

# Wait for startup and verify webhook registration
until docker logs $(docker ps -qf name=n8n-1) 2>&1 | grep -q "Editor is now accessible"; do
  sleep 3
done
docker logs $(docker ps -qf name=n8n-1) 2>&1 | grep -E "Activated|published"
# Expected: "Activated workflow 'GitHub → Dify KB Sync'"
#           "Activated workflow 'GitLab → Dify KB Sync'"

# Test webhooks are live
curl -s -X POST http://localhost:5679/webhook/rag-sync-github \
  -H "Content-Type: application/json" -d '{}' | grep -o '"message":"[^"]*"'
# Expected: {"message":"Workflow was started"}
```

```bash
# Clean up temp files
rm -f /tmp/n8n_cookies.txt
```

## Step 9 — Post-Start: Dify Admin Setup

> Dify admin setup can be done via API (preferred) or browser wizard.

### 9.1 — Create admin account (API — one-time)

Check setup status first:
```bash
curl -s http://localhost:5001/console/api/setup
# Returns: {"step": "not_started"}  → proceed below
# Returns: {"step": "finished"}     → already done, skip
```

Create the admin account:
```bash
curl -s -X POST http://localhost:5001/console/api/setup \
  -H "Content-Type: application/json" \
  -d '{
    "email": "admin@platform.local",
    "name": "Platform Admin",
    "password": "Admin@Platform2026!"
  }'
# Expected: {"result": "success"}  HTTP 201
```

Verify login:
```bash
curl -s -X POST http://localhost:5001/console/api/login \
  -H "Content-Type: application/json" \
  -d '{"email": "admin@platform.local", "password": "Admin@Platform2026!", "remember_me": false}'
# Expected: {"result": "success", "data": "<jwt>"}  HTTP 200
```

> **Alternative:** Open `http://localhost:3002` (or `https://theaitools.ca/dify/` once nginx is configured) and complete the browser wizard instead.

### 9.2 — Configure LLM provider in Dify

1. Settings → Model Providers → add your OpenAI-compatible provider
2. Set the same LLM credentials used in Vault (`platform/global/llm`)
3. Set the embedding model (used for knowledge base indexing)

### 9.3 — Knowledge bases

Knowledge bases are created automatically by the platform when a user creates a Knowledge Source in the UI. **Do not create them manually in Dify** — the workflow-service provisions them via the Dify API using credentials stored in Vault.

Each time a Knowledge Source is synced for the first time, the platform:
1. Creates a Dify dataset via API
2. Stores the resulting dataset ID and API key in Vault at `secret/platform/global/dify/{kbId}`
3. Triggers the n8n sync workflow via webhook

## Step 10 — Seed Global App Credentials (Dify + LLM)

All keys seeded here are visible and editable in the platform UI at `/rapidrag/secrets` (admin login required).

### 10.1 — Seed Dify configuration

Seeds the Dify internal URL, admin credentials, and default LLM model names used by the platform to provision knowledge bases.

```bash
VAULT_DATA_DIR=$(docker volume inspect "$(basename $(pwd))_vault_data" --format '{{.Mountpoint}}')
ROOT_TOKEN=$(sudo jq -r '.root_token' "${VAULT_DATA_DIR}/vault-init.json")

VAULT_ADDR=http://localhost:8200 VAULT_TOKEN="${ROOT_TOKEN}" \
  vault kv put secret/platform/global/dify/config \
    default_app_url="http://dify-api:5001" \
    console_email="admin@platform.local" \
    console_name="Platform Admin" \
    console_password="YOUR_DIFY_ADMIN_PASSWORD" \
    model_provider="openai_api_compatible" \
    model_api_base="https://api.fuelix.ai" \
    model_api_key="YOUR_LLM_API_KEY" \
    chat_model="claude-sonnet-4-6" \
    embedding_model="text-embedding-3-small" \
    default_api_key="" \
    dataset_api_key=""
```

> `default_app_url` — internal Docker network URL, always `http://dify-api:5001` (never the public hostname)
> `console_password` — must match the password used in Step 9.1 admin setup
> `model_api_base` — LLM provider base URL that Dify uses to configure its model providers (e.g. `https://api.fuelix.ai`)
> `model_api_key` — LLM API key that Dify uses; can be the same key as Step 10.2
> `chat_model` / `embedding_model` — must be model names supported by your LLM provider
>
> All these fields are visible and editable after deployment at `/rapidrag/secrets` (admin login required).

### 10.2 — Seed LLM Credentials (AI Prompt Generation)

The **AI Agent Prompt** page uses an OpenAI-compatible LLM to generate system prompts. Without this the "Generate" button returns 503.

```bash
VAULT_DATA_DIR=$(docker volume inspect "$(basename $(pwd))_vault_data" --format '{{.Mountpoint}}')
ROOT_TOKEN=$(sudo jq -r '.root_token' "${VAULT_DATA_DIR}/vault-init.json")

VAULT_ADDR=http://localhost:8200 VAULT_TOKEN="${ROOT_TOKEN}" \
  vault kv put secret/platform/global/llm \
    api_key="YOUR_LLM_API_KEY" \
    model="claude-sonnet-4-6" \
    base_url="https://api.fuelix.ai"

# Revoke root token after both 10.1 and 10.2 are done
VAULT_ADDR=http://localhost:8200 VAULT_TOKEN="${ROOT_TOKEN}" \
  vault token revoke "${ROOT_TOKEN}"
```

> Using OpenAI: `base_url="https://api.openai.com/v1"`, `model="gpt-4o-mini"`
> Using Anthropic directly: `base_url="https://api.anthropic.com/v1"`, `model="claude-sonnet-4-6"`

## Known Issues (Fixed in Current Codebase)

### MinIO fails to start: `'sh' is not a minio sub-command`

**Status: Fixed in docker-compose.yml**

The `minio/minio:latest` image uses `ENTRYPOINT ["/usr/bin/docker-entrypoint.sh"]` which passes all arguments to the `minio` binary. The TLS setup command starts with `sh -lc …`, which minio interprets as a subcommand. Fixed by adding `entrypoint: /bin/sh` to the minio service so the command runs as a plain shell script.

### Keycloak fails to start: `Unrecognized field "_users_note"`

**Status: Fixed in infra/keycloak/realm-export.json**

JSON does not support comments. A `_users_note` documentation field was present in the realm export; Keycloak's strict JSON deserialiser rejects any unknown fields. Removed from the file.

## Rollback Procedure

```bash
# Stop all services
docker compose -f docker-compose.yml -f docker-compose.prod.yml \
  --env-file .env.production down

# Remove nginx config
sudo rm /etc/nginx/sites-enabled/theaitools-rapidrag
sudo nginx -s reload

# Check logs for diagnosis
docker compose -f docker-compose.yml -f docker-compose.prod.yml \
  --env-file .env.production logs web
docker compose -f docker-compose.yml -f docker-compose.prod.yml \
  --env-file .env.production logs api-gateway
```

To do a full clean rebuild (wipes all data volumes):

## Architecture

```
Internet (Cloudflare → theaitools.ca)
    │
    ▼
Host Nginx (ports 80/443)  ←  Let's Encrypt TLS
    │
    ├── /rapidrag/          →  https://localhost:3443   (inner Docker nginx → Next.js)
    ├── /rapidrag/connect/  →  https://localhost:4000   (api-gateway)
    ├── /n8n/               →  http://localhost:5679    (n8n)
    └── /                   →  other paths / 404

Internal Docker network (all services communicate over TLS — certs issued by Vault PKI):
    vault  →  postgres, redis, rabbitmq, opensearch, minio, keycloak
    api-gateway  →  postgres, redis, rabbitmq, opensearch, keycloak
    workflow-service  →  postgres, redis, rabbitmq
    web  →  api-gateway (proxied via nginx)
```
